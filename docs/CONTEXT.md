# Project Context

> **READ FIRST.** This document contains the decision history and current state of the project. If a development session was interrupted, start here to resume where you left off.

## Origin

The project was conceived by a web developer father for his non-verbal autistic son (nickname "Nick"). The speech therapist recommended high-tech alternative communication, but the cost of an iPhone + AAC app subscriptions is prohibitive. The solution is a free PWA that runs on any device.

## Architectural Decisions

1. **Pure PWA, no app stores** — avoids approval, cost, and platform dependency. The app is served via HTTPS and installable from the home screen.
2. **React 18 + Vite** — AI generates high-quality React code. Vite is fast and simple.
3. **No external state frameworks** (Redux, Zustand) — useReducer + Context is sufficient for the app's complexity.
4. **IndexedDB via idb wrapper** — structured offline persistence without relying on loose JSON in localStorage.
5. **Dual TTS provider:** Native Web Speech API as default/fallback; Narrator TTS API as optional premium engine. Controlled by `ttsProvider` setting in SettingsContext.
6. **Narrator TTS via proxy:** Calls go through `/api/tts/synthesize` on the same origin. Nginx (prod) or Vite dev server (dev) adds the `X-API-Key` header — the client JS never sees the key.
7. **Audio caching:** Generated MP3 blobs are stored in IndexedDB (`audio_cache` store) so repeat taps play instantly without API calls.
8. **i18next for internationalization** — industry standard, supports automatic detection + manual switching.
9. **ARASAAC as pictogram source** — free, compatible license, thousands of standardized symbols that speech therapists already know.
10. **Responsive Grid with CSS Grid** — rows×cols defined per board, cell size calculated by `min(height/rows, width/cols)`.
11. **Admin mode via long press (5s, 3 fingers)** — non-intrusive yet accessible for speech therapists to set up the board.
12. **Vibe coding as methodology** — code is generated by AI (current high-quality models), reviewed, and versioned with git. Tags mark stable versions. If something breaks, revert to the stable tag.
13. **API key at container runtime** — `$TTS_API_KEY` read via `envsubst` in nginx CMD, never baked into Docker image. Pass as env var in `docker run` / compose on VPS.
14. **CI/CD pushes to ghcr.io** — image is key-free. Watchtower on VPS auto-updates container on push.

## Development Rules

- **Never add unnecessary dependencies.** Prefer native APIs (Web Speech API, CSS Grid, IndexedDB) over external libraries.
- **Test on the target device (Android) after every significant change.** What works on desktop may fail on mobile.
- **i18n is mandatory from the start.** Every visible string must go through i18next. No hardcoded text.
- **Performance on entry-level devices is a priority.** The app must run smoothly on a $100 Android phone.
- **Each board is an independent entity.** The speech therapist can create multiple boards for the same child.
- **Zero crash tolerance in user mode.** Admin mode may have bugs, but user mode (child using it) must be rock solid.
- **API key must never appear in client JS or Docker image** — only as env var at container runtime.

## Glossary

| Term | Definition |
|------|-----------|
| **Board** | A communication grid with rows×cols cells |
| **Cell** | An individual button with an image, label, and speech text |
| **Grid** | The visual grid rendered from a board |
| **Pictogram** | The image/symbol associated with a cell (via ARASAAC) |
| **Speech** | The text that TTS speaks when tapping a cell |
| **Admin mode** | Edit mode where the speech therapist sets up/configures boards |
| **User mode** | Usage mode where the child taps cells to communicate |
